# AgentsShop 产品需求文档（PRD）

## 1. 文档信息
- 产品名称：AgentsShop（桌宠 Agent 工作台）
- 文档版本：v1.0
- 文档日期：2026-03-04
- 适用范围：桌面端（Electron）+ 轻后端（Python Datacenter）

## 2. 产品背景
当前项目已具备“桌宠入口 + Agent 对话 + 基础管理”的雏形能力：用户可通过桌宠低打扰交互进入主控制台，与单一 Agent（Sera）进行流式对话，并查看基础状态与管理页面。  
现阶段需要将“可运行原型”收敛为“可发布的 MVP 产品”，建立清晰的功能边界、质量标准和迭代优先级，支撑后续多 Agent、权限体系和稳定性升级。

## 3. 产品目标
- 打造低打扰、可常驻的桌面 AI 助手入口。
- 提供稳定、连续的流式对话体验与会话追溯能力。
- 建立最小可用的管理能力，支持 Agent 配置与运行状态查看。
- 形成可运维、可诊断、可恢复的本地运行闭环。

## 4. 目标用户与场景
### 4.1 目标用户
- 普通用户：需要随时发起 AI 对话，但不希望被复杂界面打扰。
- 管理用户：需要管理 Agent 配置、查看运行状态和问题定位信息。
- 开发运维用户：需要快速启动、调试、清理运行环境。

### 4.2 核心使用场景
- 用户在桌面常驻桌宠，点击后打开控制台并开始对话。
- 用户在会话中获得流式回复，并可回看历史消息。
- 管理用户进入后台查看 Agent 状态并进行基础管理操作。
- 开发运维通过脚本启动与清理，快速恢复工作环境。

## 5. 范围定义
### 5.1 MVP 范围（本期必须）
- 单 Agent（Sera）接入与可用性保障。
- WebSocket 流式对话（含增量展示、完成态、错误态）。
- 消息持久化与会话级历史读取。
- 桌宠窗口、主应用窗口、状态窗口三者协同。
- 托盘常驻与基础窗口控制（显示、隐藏、退出）。
- 基础管理页与路由守卫（保留扩展点）。
- 日志记录与一键清理脚本。

### 5.2 非本期范围
- 多 Agent 协作编排。
- 云端账号体系与跨端同步。
- 复杂业务数据库与审计系统。
- 商业化支付、订阅和计费能力。

## 6. 功能需求
### FR-01 桌宠交互
- 支持桌宠常驻、拖拽、速度调节、状态反馈。
- 支持鼠标穿透与交互命中切换。
- 支持从桌宠快速打开主控制台。

### FR-02 对话能力
- 支持用户发起消息并通过 WS 接收流式回复。
- 支持消息状态：进行中、完成、错误。
- 支持首字延迟统计与基础链路日志。

### FR-03 消息与会话
- 按 Agent 维度持久化消息记录。
- 支持按会话读取历史并在前端回放。
- 支持基础数据清理策略（脚本化）。

### FR-04 管理能力
- 支持展示 Agent 列表、可用性与元信息。
- 支持进入管理后台并进行基础配置查看。
- 保留角色模型扩展接口（guest/user/admin）。

### FR-05 运行与运维
- 支持一键启动开发环境（后端、前端、Electron）。
- 支持异常时快速清理进程与日志。
- 支持关键运行日志可追踪。

## 7. 非功能需求
- 性能：首字返回体验稳定，流式过程无明显卡顿。
- 可用性：核心链路在常规本地环境可稳定运行。
- 可靠性：异常时可通过脚本快速恢复到可运行状态。
- 安全性：默认不在日志输出敏感配置；环境变量管理规范化。
- 可维护性：模块边界清晰，支持后续多 Agent 扩展。

## 8. 关键流程
### 8.1 对话主流程
1. 用户在控制台发送消息。  
2. 前端通过 WS 发送 `chat` 请求。  
3. 后端将消息转发给 Agent，并按 `delta` 流式返回。  
4. 前端增量渲染内容，接收 `done` 后写入最终状态。  
5. 消息服务落盘，供历史回放。

### 8.2 桌宠到控制台流程
1. 用户点击桌宠。  
2. 主进程接收事件并激活主应用窗口。  
3. 用户进入聊天页面继续会话。  

## 9. 指标与验收
### 9.1 核心指标
- 会话成功率：消息发送到完成态的成功比例。
- 首字延迟：发送后首次 `delta` 的耗时。
- 崩溃恢复效率：异常后恢复到可用状态的操作成本。
- 留存指标：用户重复打开与重复对话频次。

### 9.2 验收标准
- 单 Agent 对话链路完整可用，错误态可感知。
- 历史消息可按会话稳定读取与展示。
- 桌宠、主窗口、状态窗口行为一致且无阻塞。
- 启动与清理脚本可在标准开发机稳定执行。
- 管理页可访问，具备基础展示能力。

## 10. 版本规划
### V1（当前）
- 聚焦单 Agent 桌面工作台 MVP，打通可用闭环。

### V1.1
- 完成登录鉴权闭环与角色权限真实生效。
- 加强管理页配置能力与状态观测能力。

### V1.2
- 支持多 Agent 注册、切换与统一编排入口。
- 补充更完善的数据管理策略与稳定性优化。

## 11. 风险与对策
- 风险：当前权限默认 admin，存在上线安全隐患。  
  对策：优先落地登录鉴权、默认最小权限策略。  
- 风险：消息采用本地 JSONL，规模增长后读写压力上升。  
  对策：引入归档策略与后续数据库迁移预案。  
- 风险：多进程多窗口带来状态同步复杂度。  
  对策：统一事件总线规范与关键链路监控。  
- 风险：模型接口波动影响流式稳定性。  
  对策：增加重试、超时与错误分级提示机制。  

## 12. 依赖与约束
- 依赖本地 Python 运行环境、Node/Electron 开发环境。
- 依赖模型服务可用及兼容接口稳定。
- 约束于桌面端优先，本期不包含移动端与 Web 独立发布。
